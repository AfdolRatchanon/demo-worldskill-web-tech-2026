## 2. Introduction
A testing organiser is running a regional web technologies competition in an isolated local area network environment. Each competitor must build a complete full-stack web application that can be accessed from the internal room network only. Internet access is not available. The competitor is responsible for ensuring that the submitted application URLs remain accessible within the local competition network.

## 3. Scenario
You are asked to build a Test Submission Management System for a competition room. The system must allow candidates to review the task, submit application URLs, and view submission status. Judges must be able to open and close the testing session, re-check submissions, and confirm results. Managers must be able to view score summaries and ranking dashboards.

## 4. Description of project and tasks

| Role | Key responsibilities |
|---|---|
| Candidate | Log in, review the task, see countdown timer, submit Frontend URL and Backend API URL, review submission status and latest result.|
| Judge | Open/close test session, view candidate list, review submissions, trigger re-check, confirm scores. |
| Manager | View summary dashboard, ranking, pass/fail status, and export report. |

## 5. Frontend application
### Requirements
- Responsive layout for desktop and mobile view. The user interface must remain usable at standard desktop width (for example 1366px) and narrow/mobile width (for example 375px).
- Login page with proper validation and role-based redirect.
- Candidate dashboard with visible countdown timer, submission form, and latest result card.
- Judge dashboard with candidate table, session control buttons, submission list, re-check and confirm actions.
- Manager dashboard with score summary, ranking and pass/fail overview.
- Validation feedback must be visible and understandable.
- Keyboard usability and focus visibility should be preserved.

### Screens

| Screen | Minimum requirements |
|---|---|
|Login|Username/password form, error feedback, redirect according to role.|
|Candidate Dashboard|Task summary, countdown timer, submit/update Frontend URL and Backend URL, latest result.|
|Judge Dashboard|Candidate list, start/close session, submission table, re-check button, confirm result.|
|Manager Dashboard|Summary cards, ranking table, pass/fail counts, export action.|

## 6. Backend application
### Requirements
- REST API with JSON responses.
- Authentication and role-based authorization.
- Database persistence for users, candidates, session status, submissions, and results.
- Validation and business rules enforced on every relevant endpoint.
- Statistics and report endpoints for summary and ranking.

### APIs
| Method | Endpoint                              | Purpose                                              |
|--------|---------------------------------------|------------------------------------------------------|
| POST   | /api/login                            | Authenticate user and return token/session           |
| POST   | /api/logout                           | Logout current user                                  |
| GET    | /api/config                           | Return current session status and timing             |
| GET    | /api/tasks                            | Return task overview for current user                |
| GET    | /api/my-submission                    | Return current candidate submission                  |
| POST   | /api/my-submission                    | Create candidate submission                          |
| PUT    | /api/my-submission                    | Update candidate submission before session closes    |
| GET    | /api/my-result                        | Return latest grading result for current candidate   |
| GET    | /api/candidates                       | Judge list of candidates                             |
| GET    | /api/submissions                      | Judge list of all submissions                        |
| PUT    | /api/session/start                    | Judge opens the test session                         |
| PUT    | /api/session/close                    | Judge closes the test session                        |
| POST   | /api/submissions/{id}/recheck         | Judge requests auto-check again                      |
| PUT    | /api/results/{candidate_id}/confirm   | Judge confirms final score                           |
| GET    | /api/statistics/summary               | Manager summary metrics                              |
| GET    | /api/statistics/ranking               | Manager ranking data                                 |
| GET    | /api/statistics/status                | Manager pass/fail summary                            |
| GET    | /api/report                           | Manager export payload or report data                |

### Success format
``` json
{
    "success": true,
    "data": {},
    "meta": {}
}
```

### Error format
``` json
{
    "success": false,
    "message": "Error message"
}
```

7. Business requirements
- A candidate may have only one active submission for the current session.
- A candidate cannot create or update a submission before the session starts or after it closes.
- Judge users are the only users allowed to start or close the session.
- Manager users are read-only users.
- Candidates can access only their own submission and result data.
- Submitted URLs must be valid internal LAN URLs and must be reachable by the checking system.
- If the testing session is closed, protected operations must return an appropriate error response (for example HTTP 403).
- The final application must continue to work with the provided seed data, although the database
schema may be extended if required.

## 8. Accessibility
The client requires good accessibility by following WCAG 2.1 principles. Use semantic HTML, proper labels, focus visibility, sufficient contrast, and keyboard-usable controls.
The competitor is responsible for ensuring that submitted URLs remain reachable by the competition checking system throughout the assessment period.

## 10. Instructions to the competitor
- You may choose your own framework and stack, but the final application must run without internet access.
- Frontend applications must only connect to the backend via APIs requests. The frontend applications are not allowed to connect to any databases in any cases.
- The application must be accessible from the local room network.
- External APIs, external fonts, CDN assets, and cloud deployment are not allowed.
- You must keep existing seed data working and may extend the schema if required.
- Write code that is easy to maintain and clearly structured.
